## 2. 计划开发的工具清单与大体规划

### 2.1 P0（M1-A 首版建议一次交付）

目标：**一个可发布、可 SEO 的「主工作台」**，用 **Tab（或清晰分区）** 覆盖编码/解码主路径，避免多个薄内容落地页内耗（与《Base64 产品线》§2 一致）。

| 交付单元 | 用户价值 | 技术/产品要点（概要） |
| --- | --- | --- |
| **Base64 工作台 · 文本编码** | UTF-8 文本（含粘贴的 HTML/CSS/JSON 等「源码」）→ Base64；可选输出为 **Data URL 前缀** 说明。 | 纯前端 `TextEncoder` → bytes → Base64；非法 surrogate 提示；可选「仅输出纯 Base64 / 带 data: 前缀」说明区。 |
| **Base64 工作台 · 文件编码** | 任意文件（图片/音频/视频/PDF 等）→ Base64 或 **Data URL**。 | `FileReader` → `ArrayBuffer` → Base64；MIME 来自 `file.type` 或可选手动覆盖；**大文件体积上限 + 卡顿/内存风险提示**（规划文档 §1）。 |
| **Base64 工作台 · 解码** | Base64 字符串 → UTF-8 文本预览；**Data URL** → MIME 识别、文本预览或触发下载。 | 校验 padding/非法字符；非 UTF-8 二进制解码后的展示策略（下载为主 + 可选 hex 预览一句说明）。 |
| **共用能力** | 复制结果、清空、示例占位、链到全站 **隐私** 页；工具内一句 **默认不上传** 声明。 | Client 组件承载输入状态与 `clipboard`；壳层保持 Server + `ToolPageLayout`。 |

**URL / slug（建议，可评审后定稿）**

| 方案 | 说明 |
| --- | --- |
| **推荐（首版）** | **单 slug**：例如 `tools/base64` 或 `tools/base64-workbench`，**全部 Tab 同页**；SEO 用 **独立 title/description + 页内 H2** 区分意图，必要时加 FAQ 段落区分「文本 / 文件 / 解码」。 |
| **备选** | **双 slug**：`tools/base64-encode` 与 `tools/base64-decode`，canonical 互指或 301 到工作台锚点 — 维护成本更高，仅在有明确搜索词策略时采用。 |

> 功能规划表中的 `text-base64-encoder`、`file-to-base64`、`base64-decoder` 可作为 **别名/后续 301** 或 **首版即选其一作为主 slug**，在定稿段记录最终决策即可。

### 2.2 P1（首版之后，有独立文案价值再上）

| 工具 | 建议 slug | 与主工作台关系 |
| --- | --- | --- |
| **Hex ↔ Base64** | `hex-base64-converter`（见功能表 B-04） | 独立说明奇数位、非法 hex；主工作台 **「高级」入口跳转** 或第二迭代再上。 |
| **URL 拉取 → Base64** | `url-to-base64`（B-05） | **CORS / 混合内容** 说明为主；可与文件 Tab 的「从 URL 拉取」子区合并，独立 slug 仅在有长文 FAQ 时值得做。 |

### 2.3 P2（远期）

- **Base64URL**（JWT 等）、PEM 行清理、批量 ZIP 等 — 见《Base64 产品线》§3.5；不纳入 M1-A 范围。

### 2.4 开发顺序建议（实现阶段用）

1. **路由 + 文案骨架**：`tools/{主slug}`、`generateMetadata`、sitemap 登记、`messages` 中英键结构（五语言可先英后补译）。  
2. **解码 Tab**（输入校验与文本预览路径最短，利于打通布局与错误态）。  
3. **文本编码 Tab**。  
4. **文件编码 Tab**（含大小限制与说明）。  
5. **Data URL 解析与下载** 与解码 Tab 的边界收束。  
6. **复制 / 无障碍 / 移动端** 走查。  
7. **导航与首页入口**（见 §4）与 P1 入口占位（可先灰显或仅文档链接）。

## 3. 导航栏：按分类 + 悬浮下拉（你的建议）

### 3.1 形态

- 顶栏 **「主导航」** 在现有「首页」旁增加 **产品线分类**，首版至少一项：**Base64**（文案 key 建议 `nav.categories.base64` 或 `nav.base64`）。  
- **桌面**：鼠标 **悬停（hover）** 在「Base64」上时，展示 **下拉面板**（或向下展开的菜单），列出 **该分类下全部已上线工具**（首版可能只有「Base64 工作台」一条；P1 增加 Hex、URL 等后再追加行）。  
- **键盘**：分类项支持 `aria-expanded` / `aria-controls`，Enter 或 Space 展开，Esc 关闭；焦点顺序符合 WCAG 操作习惯。  
- **触屏**：**不依赖 hover**；点击分类名 **第一次展开**、再点外部或同一项关闭（与语言切换组件的交互策略可复用思路，具体实现时统一）。

### 3.2 下拉项内容（每条）

- **标题**：工具短名（与 `messages` 一致）。  
- **链接**：`hrefForLocale(locale, "tools/{slug}")`。  
- **可选**：一行副标题（用途一句话），利于扫描；注意长文案在多语言下的折行。

### 3.3 后续扩展

- 第二产品线（如 **JSON**）出现时，同样增加一个分类 + 下拉，与 **G-02「按产品线聚合」** 对齐。  
- 若分类过多，可再评估「更多」收纳；M1-A 阶段不需要。

## 5. SEO 与站点地图（规划层）

- 新 slug 上线后写入 **`src/app/sitemap.ts`** 的静态或生成列表（与 AGENTS 检查清单一致）。  
- 单页多 Tab 时：**一个 canonical**；Tab 切换尽量用 **hash 或客户端状态** 不改变 canonical（避免重复索引）；若采用 **独立 URL**，必须配置 **canonical 主次** 或合并策略（见《Base64 产品线》§2）。  
- 每工具页：`generateMetadata` 独立 **title / description**，正文避免与兄弟页大面积重复。
