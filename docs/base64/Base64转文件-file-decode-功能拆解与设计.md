## 一、按「四维度工作流」逐项落地

### 1. 功能范围：用户来这一页最想解决什么？

**用户核心诉求（建议写成页面的「一句话价值」）：**  
「我手里有一段 Base64（或网页里常见的那种 `data:…;base64,…`），想在本机浏览器里**还原成文件并下载**，不上传服务器。」

#### 建议的能力拆分

| 类型 | 建议内容 | 说明（通俗） |
|------|----------|--------------|
| **必做** | 粘贴或输入一段内容，点「解码 / 还原」后，得到**可下载的文件** | 和「文本解码」页区分开：这里重点是**文件（二进制）**，不是读 UTF-8 文章。 |
| **必做** | 支持常见输入形态：**纯 Base64**、**Base64URL**（`-` `_`）、**Data URL**（自动去掉 `data:类型;base64,` 前缀，只解后面的载荷） | 用户从别处复制来的格式很杂，能自动「剥壳」会少很多困惑。 |
| **必做** | **大小限制**（与现有「文件转 Base64」页面对齐或略放宽/略收紧均可，但必须在页面上写清楚） | 防止一次解码几 GB 把标签页卡死；限制是**产品选择**，不是技术做不到。 |
| **必做** | **解码失败时**：用**人能看懂的一句话**说明原因大类，并给出**下一步怎么做**（见下文「失败场景」专节） | 不要只弹「Error」或静默没反应。 |
| **必做** | **隐私说明**：默认强调「解码在浏览器本地完成，不把您的字符串发到我们的服务器」（若产品确实如此） | 与全站其它 Base64 工具叙事一致，利于信任。 |
| **建议做** | **一键复制**（若中间态是「仅展示十六进制/大小」等，则按需；若最终态是下载，复制可能次要） | 若流程是「先预览再下载」，可复制预览用的片段或元信息需单独想清。 |
| **建议做** | **清空**输入与中间状态（预览、错误提示、已生成的下载链接） | 和编码页习惯一致。 |
| **建议做** | **文件名**：有 Data URL 时可从 MIME 推断扩展名；纯 Base64 时可给默认名（如 `decoded.bin`）或允许用户改扩展名/文件名 | 纯 Base64 没有「官方文件名」，需要产品给一个**合理默认** + 可选改名的出口。 |
| **可选** | **小预览**：解码结果是图片时，在下载前显示缩略预览；音视频可只显示「类型 + 大小」 | 提升信心，但实现与体积限制要一起考虑。 |
| **可选** | **从本机选 `.txt` 读入**（里面是 Base64） | 对开发者友好；不做也能用粘贴覆盖大部分场景。 |

#### 建议「明确不做」（边界）

| 不做的事 | 原因（通俗） |
|----------|--------------|
| **不负责「解密」** | Base64 只是编码，不是密码学意义上的加密；本页不扮演「破解密文」工具。 |
| **不保证「解码结果一定是您心里的那种文件」** | 给什么字节就还原什么字节；若 Base64 本身是错的或截断了，只能还原出**错的或坏的**文件，工具无法替用户猜意图。 |
| **不把本页做成「万能十六进制编辑器」** | 深度编辑二进制、对比 diff、病毒扫描等，应属其它工具或本机软件。 |
| **不与「Base64 转文本」抢同一职责** | **文本解码**：强调 UTF-8 可读、段落预览。**文件解码**：强调**下载二进制 / 按类型保存**；两页标题、描述、首屏说明应让用户一眼知道去哪一页。 |

### 2. 页面结构：信息从哪进、从哪出？

**与四维度工作流对齐的结论（线框级）：**

- **宽屏：** 建议 **左输入、右输出/预览与操作**（或镜像，但全站习惯若已定左输入则对齐）。左侧：大文本框 + 主要按钮（解码、清空）+ 选项；右侧：错误提示区、元信息（解码后大小、推断的 MIME）、下载按钮、可选图片预览。

## 二、解码失败与边界场景：发生了什么、该怎么提示、怎么处理

下面用**用户能听懂**的说法列场景；实现时可以把每一类映射到**明确的错误码或错误文案模板**（本文不写代码枚举名）。

### 1. 输入为空或只有空格

- **现象：** 没有有效字符可解。  
- **处理：** 不要生成下载；提示类似「请先粘贴或输入 Base64 内容」。  
- **边界：** 不要把「空」当成合法输入去解出一堆奇怪字节。

### 2. 不是合法 Base64 字符（含截断、乱码、把 URL 当 Base64 整段贴入）

- **现象：** 字符表不对、长度不对、填充不对等，**无法完成解码**。  
- **处理：** 提示「内容不像有效的 Base64，请检查是否复制完整、是否混入了换行或说明文字」；可选：提示用户若来自 Data URL，是否应只复制 `base64,` 后面的部分（若当前解析器未自动剥离）。  
- **边界：** 不必尝试「智能修复」所有损坏数据（避免无限猜测）；若未来做「一键去除非法字符」可作为**可选**增强并明确说明副作用。

### 3. Data URL 格式识别失败

- **现象：** 用户写了 `data:` 开头但不符合 `data:…;base64,` 的常见形态。  
- **处理：** 提示「当前内容不是可识别的 Data URL，请检查前缀；或改用纯 Base64 模式粘贴」。  
- **边界：** 不必支持所有历史 MIME 写法；**文档里写清**支持哪几种写法即可。

### 4. 解码成功但长度为 0

- **现象：** 合法但载荷为空。  
- **处理：** 提示「解码结果为空，没有可下载的文件」；不提供空文件下载或提供明确命名的 0 字节文件（二选一由产品定，**建议不下载**以免用户困惑）。

### 5. 超过产品设定的大小上限

- **现象：** Base64 很长，解码后字节数超过**页面声明的上限**。  
- **处理：** **在解码前或流式处理中**检测并中止；提示「内容超过本页上限（×× MB），请分段处理或使用本机脚本」。  
- **边界：** 上限是产品策略，与「文件转 Base64」页对齐可减少用户认知成本。

### 6. 解码成功，但无法推断「该当成什么文件」

- **现象：** 纯 Base64，没有 Data URL 的 MIME；魔数（文件头）也不匹配常见图片/压缩包等。  
- **处理：** 默认提供下载，扩展名用 **`.bin` 或通用默认名`**，并在界面上写清「未识别类型，已按二进制保存；可自行改扩展名后再打开试试」。若提供「手动选类型 / 扩展名」则作为增强项写进功能表。  
- **边界：** 不保证改扩展名后一定能被某软件打开——那是文件内容与用户环境的问题。

### 7. 解码成功且像某种格式，但实际损坏（例如半截 PNG）

- **现象：** 能下载、扩展名也对，但系统图片查看器报错。  
- **处理：** 在说明文案里写一句「若文件无法打开，可能是原始 Base64 不完整或来源本身已损坏」；**不必**在页内做深度文件修复。  
- **边界：** 与「解码失败」区分：字节层面解码是成功的，**问题在内容语义层**。

### 8. 与「Base64 转文本」的交叉：解码出来其实是 UTF-8 文本

- **现象：** 用户其实想要可读文字，误进本页。  
- **处理：** 可选：若检测到**整段都是可打印 UTF-8 文本**且体积不大，给一条**非阻断**提示：「这段看起来更像纯文本，您也可以使用「Base64 转文本」页获得更好阅读体验」，并链到对应工具。  
- **边界：** 启发式检测可能误判，提示应为**建议性**，不应阻止用户下载 `.txt` 或 `.bin`。

### 9. 浏览器与权限相关

- **现象：** 下载被拦截、剪贴板拒绝访问等。  
- **处理：** 提示用户检查浏览器下载权限、弹窗拦截；复制类操作失败时提示手动选择复制（与现有工具一致即可）。
